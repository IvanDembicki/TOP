# Violation Catalog

This file defines canonical violation codes for all known TOP violations.

All validation, audit, and repair output must reference these codes.
Consistent codes enable stable cross-session reporting and precise repair targeting.

---

## Core violations — CORE-xxx

Affect architectural correctness. Always block finalization.

| Code | Name | Description |
|------|------|-------------|
| CORE-001 | Controller/Content boundary violation | Controller accesses content platform primitive directly (e.g. `this.el`, `this.getView().classList`, `content.getView()`) or treats content as a mutable command target. Controller-to-content access is lifecycle/materialization access only; presentation changes use controller state, dirty/render refresh, and content pull through `IControllerAccess`. |
| CORE-002 | Content encapsulation violation | Content object or its platform primitive exposed as public interface outside the node. |
| CORE-003 | Hidden switchable | Node reads architectural state (mode, status, lifecycle phase, openedChild, owner-held mode flag, parent/context mode, or equivalent) and changes visual representation, available behavior, hit targets, context actions, or capability availability as a result; or a switchable holder/external traversal handles active-state behavior by iterating closed state siblings or branching on mode/status instead of delegating to `openedChild`. The state may be internal to the same node or external. Must be refactored into explicit switchable unless switching is proven not to be the correct pattern. |
| CORE-004 | Behavioral coherence violation | Node registers handlers for mutually exclusive states (mouseenter/mouseleave, pointerdown/pointerup) without a single continuous action justification. Must be split into state nodes. |
| CORE-005 | Lifecycle violation | Content not destroyed on deactivate/close without an explicit retention pattern. Hidden retention is forbidden. |
| CORE-006 | Method semantics violation | `buildChildren()` used as general init method, or any lifecycle method used for a purpose outside its defined semantics. |
| CORE-007 | Non-tree relationship | Implicit cross-link between non-adjacent nodes bypassing the tree structure. Must be expressed through a connector or explicit contract. |
| CORE-008 | Missing internal access boundary | Node has separate content but `IContentAccess` or `IControllerAccess` is absent or anonymous. Both directions must be explicit, typed, and named. |
| CORE-009 | Child node encapsulation violation | Public getter or method returns a child node reference directly. External code must not hold a reference to a non-direct child. |
| CORE-010 | Phase separation violation | Constructor used as undifferentiated initialization bucket mixing content materialization, child materialization, activation, and refresh. |
| CORE-011 | Parent-owned materialization violation | Child self-mounts or self-attaches its view into the parent integration surface from within its own lifecycle methods. |
| CORE-012 | Content materialization violation | `props.contentType` is declared in spec but no separate content class exists, or a thin stub exists while platform logic remains in the controller. |
| CORE-013 | Declaration order violation | In a one-file node implementation, Content class declared before the access boundary artifact that mediates access to it. |
| CORE-014 | View access violation | `getView()` called on a node by any caller other than its direct parent. |
| CORE-015 | Logic in content violation | Locally implemented content contains conditional selection logic, output derivation, or any behavioral/architectural decision logic, or controller pushes presentation commands/state/mutations into locally implemented content. This includes deciding, deriving, branching, selecting, toggling, formatting, concatenating, hardcoding, or computing structure, class/style/token, text, icon, visibility, handler, child output, platform primitive, representation, output value, capability, orchestration, lifecycle, protocol routing, or state-driven behavior. Locally implemented content may only materialize a structurally static content shape and apply already-resolved primitive/output values from its owning controller access contract; controller must update its own state and request dirty/render/lifecycle refresh rather than command content to show, hide, update, configure, apply state, set class/style, or render with values. |
| CORE-016 | Semantic layer platform leakage | Layer B contains platform primitives, source-framework APIs, CSS/DOM/widget terms as truth, or copies source implementation details instead of semantic intent. |
| CORE-017 | Target adaptation semantic drift | Layer C or generated target artifacts change business/system meaning, introduce new behavior, or adapt source primitives without preserving semantic intent. |
| CORE-018 | Pull-based construction violation | TOP object constructed outside its architectural tree position and pushed inward instead of being born where it belongs. |
| CORE-019 | Content semantic injection | Content constructor or any public runtime/composition entrypoint receives data, callbacks, handlers, flags, state, stores, services, child components, slots, prebuilt fragments, platform child views, child view handles, child-output getter bundles, view-model objects, config/options/props-like objects, parameter bags, runtime argument sets, arbitrary props, a dummy zero-access object, a public input object/value containing anything beyond the narrow content-to-controller owner access contract, an externally assembled access bundle, or raw imported/external function/service/store references exposed directly as access methods instead of exactly one narrow owner access interface implemented by the owning controller. |
| CORE-020 | Concrete controller exposure to Content | Content is typed against, imports, stores, inspects, or downcasts to the concrete controller class instead of depending only on a narrow typed access interface. |
| CORE-021 | Pushed child composition | Child components, child nodes, child view handles, platform child views, composition entrypoints, public runtime parameters, parameter bags, config/options/props-like objects, child-output getter bundles, or prebuilt view fragments are pushed into Content from parent/root/external code. |
| CORE-022 | View child ownership violation | View constructs, imports, directly owns, or navigates child nodes/controllers instead of asking its owning controller for opaque child view handles. |
| CORE-023 | RootContext dependency injection | `RootContext` is used as a generic dependency injection container or as a channel for application data, callbacks, services, stores, child instances, view fragments, or arbitrary props. |
| CORE-024 | Spec props / runtime input confusion | TOP spec `props` are treated as runtime parameters, component fields, platform fields, render/build parameters, or another technology-specific injection/composition channel. |
| CORE-025 | Concrete content exposure to Controller | Controller stores, types, or uses concrete Content implementation directly instead of the narrow `IContentAccess` boundary, exposing implementation or platform/component methods outside the permitted content contract. |
| CORE-026 | Controller role purity violation | Controller is materialized as a content-side or platform-renderable artifact, render/build function, platform UI lifecycle object, public runtime input receiver for content composition, or otherwise becomes the node's rendered content instead of remaining a controller-only orchestration boundary. |
| CORE-027 | IContentAccess data bag misuse | `IContentAccess` is used as a carrier for view-model values, state flags, callbacks, child-output handles, presentation commands, mutation commands, or data fields that Content reads from the controller. Content must request controller-owned data/actions/output through `IControllerAccess`; `IContentAccess` is only lifecycle/materialization access or an explicit zero-contract. |
| CORE-028 | Test-covered behavior loss | A migrated TOP branch loses, weakens, or fails to represent behavior that was covered by legacy tests. This includes cases where legacy test-covered behavior was not extracted, normalized, mapped to TOP nodes/contracts, reflected in spec/prompts, or re-covered by preserved, adapted, or newly generated TOP-compatible tests. |
| CORE-029 | Node/Controller semantic runtime input | A TOP Node/Controller receives semantic data, derived facts, callbacks, services, stores, child fragments, config/options/props-like objects, parameter bags, runtime argument sets, or any public runtime input other than its parent/owner boundary. Static nodes receive only parent/context. Runtime-created branch roots may receive parent/context plus one canonical Runtime Branch Binding input: entity context reference, stable identity key, or typed immutable DTO fallback. Parent/root/external code must not tunnel state or behavior into a child Node/Controller through target runtime input. Repair must not replace this with duplicate derivation of the same shared fact from the same cross-cutting source. |
| CORE-030 | Decomposed owner access input | Content receives anything other than the owning controller instance typed only as `IControllerAccess`/target-equivalent. This includes receiving owner access members as separate runtime props/parameters, JSX attributes, named function arguments, ad hoc object literals, method bags, facade/adapters, closure bundles, or any object assembled at the render/composition call site. Correctly named getter/action methods do not make a decomposed props bag a valid `IControllerAccess`. |
| CORE-031 | Decomposed content access input | Controller receives, stores, or uses anything other than its own Content instance typed only as `IContentAccess`/target-equivalent. This includes receiving content lifecycle/materialization members as separate parameters, method bags, facade/adapters, closure bundles, concrete Content types, platform primitives, or objects assembled outside the controller/content construction boundary. Correctly named members do not make a decomposed content-access bag a valid `IContentAccess`. |
| CORE-032 | Context data injection | A TOP object is filled with data instead of attached to context. A static node, locally implemented content object, connector, or black-box boundary constructor receives data packets, flags, callbacks, config/options/props-like objects, stores, services, child views, presentation values, visibility values, style values, text values, runtime state, handlers, or arbitrary additional arguments beyond its narrow contextual reference; a runtime-created branch root receives anything beyond parent/context plus one canonical Runtime Branch Binding input; or the owner pushes those values later through setter-style post-construction calls such as apply-config, set-data, set-visible, update-text, set-style, set-callbacks, or target-equivalent mutations. Objects must pull values through contextual contracts after attachment. |
| CORE-033 | Concrete content privacy breach | Concrete locally implemented content is imported, instantiated, typed against, downcast to, inspected, stored, or called by anything other than its owning controller, or the owning controller stores/uses it as concrete content instead of `IContentAccess`/target-equivalent. Concrete content is private implementation material; public APIs may expose only controller-level values or canon-permitted opaque placement handles. |
| CORE-034 | Controller platform/content fragment output | A controller public API returns platform view fragments, render/build trees, content fragments, style/layout fragments, JSX/widget/composable fragments, animation objects, content-owned setter handles, platform mutation handles, or any non-opaque content implementation object. Controllers are not renderers; child output handles, where allowed, are opaque and placement-only. |
| CORE-035 | Content-owned setter bridge | Content-owned setters or mutation handles cross the content boundary by being stored in controllers, returned through controller APIs, passed through `IControllerAccess`/`IContentAccess`, captured by adapters/helpers, or used as a controller-to-content update channel. Presentation changes must use controller-owned state, dirty/render/lifecycle refresh, and content pull of already-resolved values. |
| CORE-036 | Public wrapper around concrete content | A public TOP node/controller artifact is only a target-framework wrapper around concrete content, or external/parent/adapter code can name or instantiate the wrapper to reach concrete content. The runtime TOP tree is a tree of controllers, not content objects or public content wrappers. Public wrappers that forward target runtime calls into private content are concrete content exposure even when they use TOP-shaped names. |
| CORE-037 | Controller is not a runtime tree node | A file, class, function, module, or object is named or treated as a TOP controller but does not participate in the runtime controller tree. Signals include no parent/context for a non-root static node, no root/host context for a root node, no child ownership/registration, no children access, no lifecycle or inherited lifecycle, no declared child policy or explicit leaf declaration, no child-controller construction for declared children, no relationship to spec tree position, or a controller-shaped service/helper with methods but no runtime node base/interface. A controller without tree position is not a TOP controller. |
| CORE-038 | Invalid switchable holder contract | A node is modeled or implemented as switchable but has no state/candidate child, treats missing/null `openedChild` as a normal runtime case, uses nullable opened-child fallback for active behavior, has no default/initial selected child, or mixes switchable state-holder semantics with a generic runtime/library collection child policy. A valid switchable holder has at least one state/candidate child and exactly one non-null opened child; no-result answers belong to the opened state child, not to missing `openedChild`. |
| CORE-039 | External propagation preflight | A caller, dispatcher, holder, or traversal mechanism uses ask-then-handle or capability preflight to steer downward propagation instead of invoking the node boundary and letting the receiving node decide. Signals include `canHandle`, `hasCapability`, `isInteractive`, `supportsEvent`, `listensTo`, mode/status/state-child inspection, or similar probes used by external code to choose internal children, skip closed branches, or route active behavior. The caller may invoke the declared handler/query; the node owns result, no-result, no-op, stop, active/selected-child delegation, and connector delegation. |

Data-content clarification:
- Calling an architecturally allowed data node controller domain method such as
  `setAge(value)`, `updateName(value)`, or `replaceRecord(record)` is not a
  controller-to-presentation-content push.
- The data node controller may validate and mutate its own private data content
  through an internal storage boundary.
- External direct mutation of data content, presentation content direct access
  to data content, constructor data injection, and post-construction setter
  injection into child/content objects remain violations.

## Skill convention violations — CONV-xxx

Violations of skill-specific conventions. Do not automatically classify as CORE.

| Code | Name | Description |
|------|------|-------------|
| CONV-001 | Naming convention mismatch | Node, file, or class name does not follow the naming rules defined in `rules/naming-conventions.md`. |
| CONV-002 | File layout mismatch | Generated artifact not placed in the directory specified by `props.dir` or the structural correspondence rule. |
| CONV-003 | Structural Correspondence Rule violation | Prompt path does not reflect the same semantic position in the tree as the corresponding code path. See `references/artifact-layout-and-branch-derivation.md`. |
| CONV-004 | Missing prompt file | A code-generated node has no implementation prompt file in `top/prompts/`. |
| CONV-005 | Spec storage violation | Spec tree not stored as `.json` inside `top/`. |
| CONV-006 | Missing props.contentType | Node has content and its type must be explicitly recorded, but `props.contentType` is absent from the spec. |
| CONV-007 | Noncanonical TOP spec placement | A new migration branch spec is stored as an ad hoc root-level file under `top/` instead of under `top/specs/`, or implementation prompts exist without a matching canonical spec path, and no established project-local TOP convention justifies it. |
| CONV-008 | Missing TOP implementation source root | Specs or implementation prompts declare future materialization, but the branch has no declared/prepared implementation source root such as `top_src/<branch-id>/`, no placeholder for an empty root, or inconsistent `props.dir` / Expected Materialization roots. |
| CONV-009 | Noncanonical TOP spec shape | A project TOP spec uses ad hoc node identity fields such as `id`/`name` as the primary node-shape mechanism instead of canonical `type` or a project-approved equivalent, or otherwise cannot be regenerated as a TOP node tree. |
| CONV-010 | Generated layout/topology mismatch | Generated implementation paths do not mirror the approved TOP tree through `top_src/<branch-id>/`, effective `props.dir`, and prompt layout, or a semantic subtree is flattened into one folder without explicit materialization rationale. |

## Workflow gaps — WF-xxx

Violations of the execution process. Affect pipeline validity, not TOP Core itself.

| Code | Name | Description |
|------|------|-------------|
| WF-001 | Missed pipeline stage | A required stage for the active `task_mode` was skipped. |
| WF-002 | Missing output contract field | Agent output is missing a required field defined in the corresponding output contract. |
| WF-003 | Unresolved ambiguity passed through | Critical ambiguity was not resolved before the pipeline continued past the point where it was required. |
| WF-004 | Invalid Tier classification | Declared Tier does not match the actual scope of the task. Underclassification is a violation. |
| WF-005 | Invalid task_mode routing | Task was routed through the wrong mode pipeline. |
| WF-006 | Spec/code drift not checked | Changes to `src/`, generated artifacts, JSON specs, or prompts were made without running a drift check. |
| WF-007 | Repair cycle limit exceeded | `MAX_REPAIR_CYCLES = 3` reached without convergence and without escalation to re-modeling. |
| WF-008 | Missing semantic interpretation stage | Generation pipeline reached target adaptation or generation without a valid Semantic Interpreter output. |
| WF-009 | Missing target adaptation stage | Generation pipeline reached Generation without a valid Target Adaptation output for the active target. |
| WF-010 | Missing behavior preservation pass | A migration scope has legacy tests or executable behavior evidence, but Behavior Preservation Agent did not run and no Behavior Preservation Plan was produced before modeling, generation, validation, or final audit. |
| WF-011 | Core deviation marked as pass | Validation or Final Audit reports `pass`, `ready`, or `ready_for_use` while confirmed core violations or accepted core deviations remain. A documented migration waypoint may be tracked, but it cannot be converted into a passing TOP validation result. |
| WF-012 | Ad hoc core deviation acceptance | A project prompt, migration status file, validation report, repair report, or final audit labels a confirmed core violation as accepted/temporary/deferred/waypoint without a TOP-canon-defined migration waypoint for that violation. Documentation may track the violation, but it cannot convert it into an accepted deviation or repair. |
| WF-013 | Missing migration materialization plan | A migration/modeling handoff creates specs/prompts, Expected Materialization, or a generation next step without declaring canonical artifact paths, preparing the implementation source root, and recording an honest phase status such as analysis-only/modeling-only/materialization-pending. |
| WF-014 | Missing migration plan | A migration-mode task creates or changes TOP artifacts without a current branch-scoped `top/migration/<branch-id>/MIGRATION_PLAN.md` that records scope, phases, responsible agents, artifacts, gates, and rollback/stop points. |
| WF-015 | Missing or stale migration log | A migration-mode agent changes artifacts or hands off to another stage without appending an entry to `top/migration/MIGRATION_LOG.md`, or the log is rewritten instead of append-only. |
| WF-016 | Missing or stale migration workflow tree | A migration-mode task creates or changes TOP artifacts without a current branch-scoped `top/migration/<branch-id>/MIGRATION_WORKFLOW.json`, or the workflow JSON disagrees with the migration plan/status/log about scope, phase order, current phase, gates, or responsible agents. |
| WF-017 | Missing migration decomposition review | Migration modeling treats a user-named scope as a final node boundary, keeps a single-node/giant-node wrapper without recursive decomposition proof, omits candidate classification for hidden state/data/workflow/modal/form/list/helper structures, or uses `PanelDisplayStyle`/display-token clusters as a substitute for node/state decomposition. |
| WF-018 | Undisciplined accepted deviation | An accepted migration deviation lacks exact affected locations, temporary acceptance rationale, non-blocking rationale for the phase, target repair direction, expiry condition, owner phase, or canonical-target permissibility. |
| WF-019 | Migration workspace scope violation | A migration-mode agent modifies files outside the active branch workspace without an explicitly allowed thin adapter/integration reason recorded in the migration log; overwrites another branch's `top/migration/<branch-id>/` plan/workflow/reports, prompts, spec, or generated source; rewrites shared `top/migration/MIGRATION_LOG.md` instead of appending; or updates shared `MIGRATION_STATUS.md` by erasing previous branch history. The branch-owned workspace is `top/specs/<branch-id>.json`, `top/prompts/<branch-id>/**`, `top/migration/<branch-id>/**`, `top/assets/**`, `top/semantic/**`, and `top_src/<branch-id>/**`; shared migration files are append/history-preserving only. |
| WF-020 | Missing migration checkpoint | A migration-mode workflow skips or fails to persist a required checkpoint artifact for infrastructure, scope/decomposition, model/spec, canon precheck, generation, post-generation validation, repair, or final audit before handoff. |
| WF-021 | Non-independent validation | Validation or final audit relies on generator/repair memory, previous chat context, self-check claims, or prior reads instead of re-reading current skill rules and target artifacts in an adversarial pass. Generator self-check is not final validation evidence. |
| WF-022 | Migration without dedicated git branch | Migration writes were performed before creating or switching to a dedicated migration branch, branch safety was not logged, branch name does not match the migration branch id, unrelated work was mixed into migration output, remote push occurred without explicit user request, or local commit occurred without explicit request or documented commit phase. Every migration must run on a dedicated branch such as `top-migration/<branch-id>` before artifacts or source files are written. |
| WF-023 | Generator self-validation claim | A generation, repair, modeling, migration, or implementation agent claims `TOP-clean`, `CORE-015 clean`, `canon compliant`, `validation passed`, `no violations`, `ready_for_manual_QA`, `ready_for_use`, `final_status: pass`, or any equivalent validation verdict for its own artifacts. Executors produce artifacts and mechanical self-check evidence; validators produce verdicts. |
| WF-024 | Contaminated validation context | Validation or canon precheck treats prior generator/repair/modeling reports, migration log commentary, chat history, or self-check claims as proof instead of independent evidence. Previous agent reports are claims to inspect, not validation evidence. |
| WF-025 | Validation without artifact evidence | A validation PASS omits the artifacts reviewed, files inspected, checks performed, canon rules checked, search/detection patterns used, artifact types reviewed, violation/no-violation evidence, or unresolved ambiguities. A verdict without artifact evidence is invalid. |
| WF-026 | Final audit accepted unproven validation | Final Audit reports pass/readiness without auditing that validation ran after generation/repair, inspected current artifacts, listed files and invariants checked, rejected generator self-validation claims, and closed all rejection tickets. |
| WF-027 | Missing rejection log entry | A failed validation does not create a structured rejection ticket or does not append the rejection entry to `top/migration/MIGRATION_LOG.md`. Failed validation creates a repair obligation and must be traceable. |
| WF-028 | Missing generator learning ledger update | A validation rejection or repair attempt does not update/read the branch-local `top/migration/<branch-id>/GENERATOR_LEARNING_LEDGER.md` before the next generation or repair attempt. |
| WF-029 | Repeated rejected strategy | A generation or repair agent repeats a strategy already rejected for the same branch/violation without an explicit validator-approved reason. Rejected strategies become negative constraints. |
| WF-030 | Repair circuit breaker exceeded | A validation gate exceeds `max_repair_attempts_per_validation_gate: 3` or the same violation repeats more than `max_same_violation_repeats: 2` without escalation to human review or top-skill rule update. |
| WF-031 | Self-certified or falsely certified delivery | Delivery is declared `complete` by the same pass that generated, migrated, synchronized, or repaired the artifacts; delivery completion lacks runner-enforced execution isolation, hard-check-verified validation evidence, a valid independent judicial handoff artifact, checked files, commands/searches run, violation classes checked, failures, or unverified areas; a required gate is `fail` or `not_verified`; protocol-only execution is reported as `runner-enforced`; or a reporting/public-record pass upgrades blocked/partial/not-verified judicial findings to complete. |

## Usage

Validation Agent and Final Audit Agent must reference violation codes in all output.

Format: `[CODE] Short description of the specific instance.`

Example: `[CORE-003] TreeItem reads isEditMode from parent and hides/shows drag handle based on it.`
